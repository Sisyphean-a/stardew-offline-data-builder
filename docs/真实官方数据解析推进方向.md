# 真实官方数据解析推进方向

草稿状态：用于后续在新对话中作为 goal 输入基础。

## 1. 目标

把当前主要依赖测试夹具格式的构建器，推进到能够基于用户自己的真实《星露谷物语》游戏目录生成真实可用数据包的状态。

最终的数据来源策略调整为：

- 官方游戏解包数据作为主数据源
- `stardew-valley-data`（E:\github\stardew-valley-data-main） 作为参照、补充和校验来源
- 人工维护文件继续作为最高优先级修正层

## 2. 总体方向

后续工作的重点，不再是继续扩展夹具专用格式，而是转向真实官方数据结构适配。

整体方向分为两条主线：

- 主线 A：建立真实官方数据解析能力
- 主线 B：保留社区数据作为补充与参照能力

两条主线最终汇合到统一的标准化实体模型和统一构建产物中。

## 3. 官方数据路线

真实官方数据应当成为主入口。

后续需要逐步建立对真实官方解包文件结构的识别、分类和提取能力，使系统能够直接从真实游戏文件中获得：

- 稳定 ID
- 官方名称
- 官方描述
- 官方分类或原始属性
- 真实图片定位信息
- 可追踪的数据来源信息

这一步的重点不是一次性覆盖所有数据类型，而是先把真实格式读通，再逐类扩展。

## 4. 社区数据路线

`stardew-valley-data` 不再被视为唯一结构化来源，而是承担三个作用：

- 参照物：帮助判断官方解析结果是否合理
- 补充层：在官方缺少结构化字段时补充可用数据
- 校验层：用于检查实体覆盖率、匹配率、缺失情况和异常结果

社区数据始终处于辅助位置，不能覆盖官方已明确提供的事实。

## 5. 数据优先级原则

后续方向继续坚持清晰的数据优先级：

- 人工修正
- 官方真实数据
- 官方本地化文本
- 社区结构化数据

其中最重要的变化是：官方真实数据需要从设计目标变成真正可执行的主数据源。

## 6. 分阶段推进方向

建议后续 goal 按阶段推进，而不是一次性追求全量完成。

### 第一阶段方向

先完成真实官方四大类基础闭环：

- 物品
- 作物
- 鱼类
- 村民

目标是让这四类数据在真实游戏目录下不再产出空包，而是能稳定生成非空数据库和数据包。

### 第二阶段方向

在四大类真实解析稳定之后，再把社区数据接回真实流程中，完成：

- 稳定匹配
- 补充字段合并
- 覆盖率检查
- 未匹配报告优化

目标是让“真实官方解析 + 社区补充”成为稳定的正式构建路径。

### 第三阶段方向

再逐步推进扩展类型：

- 礼物偏好
- 日程
- 料理
- 制作配方
- 商店
- 任务
- 收集包
- 怪物与掉落
- 矿物
- 武器
- 戒指
- 成就
- 特殊订单
- 姜岛相关数据

目标是把当前“通用识别”推进为“真实可用解析”。

### 第四阶段方向

最后再收口图片链路，让图片系统与真实官方资源结构完全对齐，形成真实可交付的图片产物。

## 7. 成功标准

这个方向完成后，项目应达到以下结果：

- 能直接读取用户真实游戏目录
- 能产出非空、可检索、可追踪来源的正式 SQLite 数据库
- 能生成可分发的 `.svdata` 数据包
- 官方数据成为主来源
- 社区数据成为补充和校验来源
- 构建报告能够真实反映缺失、未匹配和风险
- 不再依赖测试夹具格式才能得到有效结果

## 8. 非目标

后续这条路线不应把精力放在以下方向：

- 不优先做 Android 工程
- 不优先追求一次性覆盖所有类型的全部细节
- 不把社区数据直接当成唯一真相源
- 不继续围绕夹具专用格式扩展主流程能力

## 9. 最终成果描述

最终希望得到的是一个真正面向真实游戏数据的离线构建器：

- 输入是用户自己的正版游戏目录
- 可选输入是本地社区结构化数据目录
- 输出是完整数据库、数据包、报告和图片资源
- 数据以官方内容为基础，以社区信息为增强
- 构建过程可重复、可诊断、可扩展
