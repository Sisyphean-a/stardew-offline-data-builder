# 真实官方数据解析架构

## 目标

构建器以用户本机官方游戏资产为唯一游戏事实来源。官方结构数据、本地化、图片和支持资产经过
解析与跨表关联后，形成稳定、可追溯的离线数据包。

## 数据层次

### 实体资产

`Data/*.json` 和 `Characters/schedules/*.json` 中的受支持资产生成 `RawEntity`，再按官方
source ID 合并中英文记录并标准化为 `NormalizedEntity`。

### 本地化资产

`Strings/*.json` 与 `Strings/*.zh-CN.json` 解析 `LocalizedText` 引用。旧式斜杠记录通过
明确字段位置恢复展示名、描述和关联 ID。

### 支持资产

以下文件不直接创建实体：

- `Data/Shops.json`
- `Data/Locations.json`
- `Data/FishPondData.json`
- `Data/Machines.json`

它们在标准化之后通过合格物品 ID、稳定 source ID 和官方 `ContextTags` 关联到现有实体。

## 派生规则

派生结果统一写入 `extra_json.officialDerived`：

- 作物：种子、季节、成长阶段、再生、收获和种子商店；
- 鱼类：难度、行为、尺寸、时间、季节、天气、地点、鱼塘和机器用途；
- NPC：生日、地区、性别和婚恋属性；
- 物品：售价、可食用值、上下文标签、商店、机器和配方用途；
- 配方：原料和产物。

每个关联使用官方结构值，不执行游戏条件表达式，不推测运行时状态。条件原文随引用保存，交由
消费端按需要解释。

## 来源追踪

实体初始来源和参与 enrich 的支持文件都写入：

```text
extra_json._provenance.official
```

`source-discovery.json` 将支持资产标记为 `supporting`，未支持的官方文件标记为
`unrecognized`，两者不得混淆。

## 不做的推断

以下内容不从原始事件脚本或游戏机制猜测：

- 自然语言事件摘要；
- 攻略建议；
- 主观分类；
- 仅存在于编译后游戏逻辑、且官方内容资产没有稳定表达的计算规则。

这类字段缺失是明确的数据边界，不使用硬编码快照填充。

## 验收

- 游戏目录是唯一外部数据输入；
- 四大基础类型和扩展类型非空；
- 官方中文缺失报告准确；
- 支持资产可产生实际关联；
- 产物不包含外部来源 provenance；
- SQLite、图片、报告和 `.svdata` 可由真实游戏目录独立生成。
